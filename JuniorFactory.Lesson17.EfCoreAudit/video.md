## Раскладка

### 00:02 [Введение: аудит изменений в EF Core](https://www.youtube.com/watch?v=KClfbWb8VzY&t=2s)

- Забыл включить микрофон, потратил минуту – дубль второй.
- Тема – аудит запросов к базе в EF Core, платный заказ от зрителя.
- Есть два прикольных пути, оба сегодня попробуем рассмотреть.

### 00:49 [Два пути: логировать запросы или аудировать сущности](https://www.youtube.com/watch?v=KClfbWb8VzY&t=49s)

- Первый путь – просто логировать insert/update/select через DbContext в файл или консоль, простая конфигурация.
- Второй путь – аудит сущностей через ChangeTracker: видно, что было до и что стало после.
- На Core логирование сам ещё не делал – раньше было событие на Database.Log, тут посолиднее.

### 02:11 [Подключаем логирование к DbContext, чиним студию](https://www.youtube.com/watch?v=KClfbWb8VzY&t=131s)

- Лезем в ApplicationDbContext, OnModelCreating, configuration.
- Студия тупит после чистки папки с нагетами – лечится выкидыванием проекта и обратным добавлением (ну или удалить bin/obj).

### 04:24 [LogTo в консоль и первый запуск](https://www.youtube.com/watch?v=KClfbWb8VzY&t=264s)

- Контексты раскиданы по дефинишенам, цепляем LogTo для начала прямо в консольку.
- Запускаем app просто убедиться, что оно вообще пишет.

### 05:20 [Убираем авторизацию, ловим запросы](https://www.youtube.com/watch?v=KClfbWb8VzY&t=320s)

- Метод по авторизации отваливается – делаем анонимный, чтоб не логиниться.
- Видим, что запросы реально идут – примерно в таком виде они и будут писаться в лог.

### 07:00 [User ID и хардкод единички для отладки](https://www.youtube.com/watch?v=KClfbWb8VzY&t=420s)

- User ID берётся из Environment в мидлварке, без него метод ругается «идентификатор не указан».
- Чтобы не мешало, на время отладки хардкодим единичку (в коммит, понятно, не пойдёт).
- Заодно ловим апдейты от фонового сервиса, который под капотом плодит регулярные операции.

### 08:52 [Логирование в файл через NLog](https://www.youtube.com/watch?v=KClfbWb8VzY&t=532s)

- Хотим не в консоль, а в файл – ковыряем NLog и его конфиг.
- В конфиге смотрим, куда пишется лог – в папку logs рядом с бинарником.

### 10:35 [Запрос в консоли и параметры-вопросики](https://www.youtube.com/watch?v=KClfbWb8VzY&t=635s)

- В консоль плюнуло: достать из таблицы машинки все неудалённые, где User ID равен какому-то вопросику.
- Параметры идут вопросиками – это потом пофиксим, чтоб видеть реальные значения.

### 11:24 [Пытаемся настроить правила NLog](https://www.youtube.com/watch?v=KClfbWb8VzY&t=684s)

- Надо расширить правила в конфиге по имени логгера (Microsoft.EntityFrameworkCore.Database.Command).
- Спрашиваем у чата/нейронки – выдаёт какую-то дресню, в файл упорно не пишет.

### 15:23 [EnableSensitiveDataLogging и логгер из DI](https://www.youtube.com/watch?v=KClfbWb8VzY&t=923s)

- Решаем сделать свой logger provider по-цивильному.
- EnableSensitiveDataLogging убирает вопросики – параметры пишутся реальными значениями.
- LoggerFactory прокидываем через UseLoggerFactory и DI.

### 19:41 [ConfigureServices против Configure](https://www.youtube.com/watch?v=KClfbWb8VzY&t=1181s)

- Долго не мог достать логгер: в билдере сервисы ещё не собраны.
- Дошло – в Configure уже есть собранные сервисы, оттуда GetRequiredService и берём.

### 22:29 [Колхозный логгер из говна и палок](https://www.youtube.com/watch?v=KClfbWb8VzY&t=1349s)

- В файл по-нормальному всё равно не пошло – лепим по-бомжатски.
- LogTo принимает Action: берём строку и через System.IO.File.AppendAllText сами дописываем в файлик.
- По-колхозному, зато работает.

### 24:07 [Что пишется в лог: запросы, параметры, миграции](https://www.youtube.com/watch?v=KClfbWb8VzY&t=1447s)

- Теперь в логе есть запросы и параметры (User ID = 1, тот самый захардкоженный).
- Помимо запросов сыпется системка: connection открылся/закрылся, проверка миграций, операции от фонового шедулера.
- Такие логи и сама база умеет писать – но это уже немного другой подход.

### 26:43 [Второй путь: ChangeTracker и override SaveChanges](https://www.youtube.com/watch?v=KClfbWb8VzY&t=1603s)

- Первый подход откатываем, создаём отдельную ветку.
- EF через ChangeTracker отслеживает изменения моделей: кто поменялся и что хочет сделать.
- Оверрайдим SaveChanges – внутри доступен ChangeTracker, на нём и строим свой аудит.

### 29:13 [Идеология аудита: три таблицы](https://www.youtube.com/watch?v=KClfbWb8VzY&t=1753s)

- На работе делал похожее не раз: смысл один, реализации разные.
- Таблица AuditEvent (id, user id, дата) – одна операция, в ней меняется несколько сущностей.
- AuditEntities (что за сущность) и её свойства old/new – тип сущности храним циферкой, чтоб место экономить.

### 32:19 [Обход Entries: old/new значения](https://www.youtube.com/watch?v=KClfbWb8VzY&t=1939s)

- Идеологию хочу донести, табличек по-настоящему не создаём.
- foreach по ChangeTracker.Entries: для каждого свойства OriginalValue и CurrentValue.
- На удаление/вставку нужен switch: при удалении заполнен Original, при вставке – Current.

### 34:32 [Меняем две сущности и смотрим результат](https://www.youtube.com/watch?v=KClfbWb8VzY&t=2072s)

- В тестовом методе берём первую машину и меняем ей имя.
- Чтобы было нагляднее, заодно правим вторую сущность – операции прибавляем к сумме единицу.
- Дёргаем SaveChanges, запускаем.

### 36:44 [Entries в дебаге: CurrentValues и Metadata](https://www.youtube.com/watch?v=KClfbWb8VzY&t=2204s)

### 38:26 [Имена свойств через Metadata и сравнение значений](https://www.youtube.com/watch?v=KClfbWb8VzY&t=2306s)

- Имя свойства достаём из Property.Metadata.Name, значения – из CurrentValues/OriginalValues.
- Логируем только то, где OriginalValue не равно CurrentValue – но это объекты, сравнение по типам надо тестировать.

### 40:03 [Кастомные атрибуты и скрытие чувствительных полей](https://www.youtube.com/watch?v=KClfbWb8VzY&t=2403s)

- User ID прокидываем в аудит из контекста (Environment), возможно сам Save дёргать в конце операции в мидлварке.
- Для UI вешаем кастомные атрибуты: человекочитаемые имена таблицы и колонки.
- Нет атрибута – таблицу не аудируем; через рефлексию собираем словарь один раз и кэшируем; чувствительное (паспорт, токен) прячем отдельным атрибутом.

### 42:22 [Фильтрация работает: два механизма аудита](https://www.youtube.com/watch?v=KClfbWb8VzY&t=2542s)

- Значения приводим к строке – в базе всё равно строкой хранить.
- Фильтрация сработала: у машины поменялось имя, у операции сумма 217 → 218.
- Так в EF Core познакомились с двумя механизмами аудита.

### 44:22 [Реальный аудит с работы: атрибуты](https://www.youtube.com/watch?v=KClfbWb8VzY&t=2662s)

- Показываю кусок боевого кода (работу стримить нельзя, потому скриншотами).
- Audit-атрибут помечает аудируемые сущности, отдельные атрибуты прячут чувствительные поля.

### 47:29 [Enum состояний и помеченный класс](https://www.youtube.com/watch?v=KClfbWb8VzY&t=2849s)

- Одна сущность может маппиться на несколько табличек.

### 49:01 [Три таблицы ревизий и метод CheckRevision](https://www.youtube.com/watch?v=KClfbWb8VzY&t=2941s)

- С первой работы: при сохранении дёргается CheckRevision, рефлексия по атрибутам подкэширована.
- Таблицы ItemChange, EntityRevision (кто, когда, что) и EntityRevisionItem – те самые три таблицы.
- Если ничего не поменялось – ревизию не создаём; для Guid и связей один-ко-многим свои костыли.
